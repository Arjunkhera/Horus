## Critical Rules — read every time

1. **Horus is ONE product, ONE repo.** Anvil, Forge, and Vault are package names inside this pnpm monorepo. They share the git repo, Typesense instance, Neo4j instance, and CI pipeline. They cannot be developed, tested, or deployed independently. There is no separate `Anvil` or `Forge` or `Vault` repository.

2. **Never edit `docker-compose.yml` to fix a real bug.** The repo's `docker-compose.yml` is the **CI publishing manifest** — it does not run locally. Users get a different `docker-compose.yml` inside their workspace directory, **generated by the CLI from the template at `packages/cli/compose/docker-compose.yml`**. Hand-editing either of those is a footgun:
   - Edits to the workspace's compose file are silently overwritten the next time `horus setup` runs.
   - Edits to the repo's compose file affect CI publishing, not user experience.
   The fix for service-runtime bugs lives in the **CLI's compose template** (`packages/cli/compose/docker-compose.yml`) and ships via a CLI release.

3. **Never use `docker compose` directly to run Horus.** All user-facing operations go through the `horus` CLI (`horus up`, `horus down`, `horus status`). Local image building creates confusion with GHCR images.

4. **Images are published to GHCR, pulled at runtime.** Push to `master` → CI builds affected images → `ghcr.io/arjunkhera/horus/<service>` updated → users get new images on next `horus up`. There is no local image building in the user-facing flow.

5. **Edit code through `forge_develop` sessions, not the canonical clone.** A workspace-level guard hook blocks Edit/Write/Bash on the canonical clone path. Use `forge_develop({ repo: "Horus", workItem: "<id>" })` to get an isolated worktree. All edits go inside the returned `sessionPath`.

### Where edits do NOT belong

| Path | Why |
|---|---|
| Repo root `docker-compose.yml` | CI publishing manifest only — does not run for users. |
| Workspace-generated `docker-compose.yml` (under `$HORUS_DATA_PATH/workspaces/<name>/`) | Generated by `horus setup` from the CLI template. Hand-edits get overwritten. |
| The canonical clone of the Horus repo | Blocked by the workspace guard hook. Use `forge_develop` for a session worktree. |

## Workspace dependencies (workspace:* imports)

```
@horus/search ← packages/anvil, packages/forge/packages/core
@forge/core   ← packages/forge/packages/mcp-server, packages/forge/packages/cli
```

Changes to `@horus/search` require rebuilding Anvil and Forge.

## Build & test (quick reference)

```bash
pnpm install && pnpm build && pnpm test     # All TypeScript packages
cd services/vault && pytest                   # Vault (Python)
cd packages/cli && pnpm build && pnpm test && npm publish  # CLI release
```

For deeper guidance, load `shared/guides/horus-development-workflow.md`.
