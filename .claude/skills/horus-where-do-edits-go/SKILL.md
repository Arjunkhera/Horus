# horus-where-do-edits-go

## When this skill fires

Use this skill the moment any of the following is true:

- The user says "edit `docker-compose.yml`" without naming which one.
- The user wants to fix a service-runtime issue (port, volume, healthcheck, env var) and you don't yet know whether the fix lives in the CLI template or the source repo's CI manifest.
- The path you're about to edit is under `$HORUS_DATA_PATH/workspaces/<name>/`.
- The path you're about to edit is under the canonical clone of a source repo — this WILL be blocked by the guard hook. Use `forge_repo_resolve` to find the canonical clone path if you need to confirm.
- You are about to write a file but haven't established whether you're inside a `forge_develop` session.

If none of the above apply, you don't need this skill.

## The three locations

| Location | Path pattern | What it is |
|---|---|---|
| **Workspace** | `$HORUS_DATA_PATH/workspaces/<name>/` | Agent context container — installed skills, `forge.yaml`, `.claude/` config, generated `docker-compose.yml`. |
| **Source repo** | Resolve via `forge_repo_resolve({ name: "<repo>" })` | Canonical clone. Read-blocked and write-blocked by the workspace guard hook. |
| **Session** | `$HORUS_DATA_PATH/sessions/<id>-<slug>/` | `forge_develop`-created git worktree. The ONLY place to edit source-repo code. |

## Decision rules

### Rule 1 — `docker-compose.yml`

There are three different `docker-compose.yml` files. Pick correctly.

| File | Edit it? | Why |
|---|---|---|
| `<source-repo>/docker-compose.yml` | **No** (and blocked) | This is the **CI publishing manifest**. It does not run for users. Editing it changes CI behavior, not user behavior. |
| `<workspace-root>/docker-compose.yml` | **No** | Generated by `horus setup` from the CLI template. Hand-edits are silently overwritten on next setup. |
| `<session>/packages/cli/compose/docker-compose.yml` | **Yes, in a session** | This is the **CLI's embedded template** — what users actually get. Edits ship via a CLI release (`npm publish`). |

**The fix path for any service-runtime bug is: `forge_develop` → edit `packages/cli/compose/docker-compose.yml` → bump CLI version → publish CLI.**

### Rule 2 — Source-repo code

Any file inside the canonical clone of a source repo is blocked by the workspace guard hook. To edit, do this:

```
forge_develop({ repo: "<repo-name>", workItem: "<your-work-item-id>" })
```

Returns a `sessionPath` under `$HORUS_DATA_PATH/sessions/`. All edits go there.

### Rule 3 — Workspace config

These edits are fine and don't need a session:

- `<workspace-root>/forge.yaml`
- `<workspace-root>/.claude/settings.local.json`
- `<workspace-root>/.claude/settings.json` (committed only if the workspace itself is a git repo)
- `<workspace-root>/.claude/scripts/*.sh`
- `<workspace-root>/CLAUDE.md`
- `<workspace-root>/workspace.env`

### Rule 4 — In-repo skill / hook (this pilot)

Files like `<repo>/.claude/skills/<name>/SKILL.md`, `<repo>/.claude/scripts/<name>.sh`, `<repo>/.claude/settings.json`, `<repo>/CLAUDE.md` live with the repo's git history. Edit them in a `forge_develop` session, like any other source-repo code. Do not put repo-specific skills in the Forge registry.

## Quick checklist before any Edit/Write

1. Is the target path inside the canonical clone of any source repo? → STOP. Use `forge_develop`.
2. Is the target a `docker-compose.yml`? → Apply Rule 1.
3. Is the target under a workspace directory (`$HORUS_DATA_PATH/workspaces/<name>/`)? → Workspace edit, fine if it's not the generated compose file.
4. Is the target under a session worktree (`$HORUS_DATA_PATH/sessions/<id>/`)? → Session edit, fine.
